# todly 개선 제안서 (Improvement Proposals)

> 각 항목은 **문제 → 제안 → 근거 → 우선순위/난이도**로 정리했습니다.
> 우선순위: P0(출시 전 필수) · P1(초기 개선) · P2(차기). 난이도: 상/중/하.

## ⚠️ v2.0 재분류 — 실제 디자인(`Tudly.dc.html`, 19개 화면) 확인 결과
v1.0에서 "차기 제안"으로 적었던 일부 기능이 **이미 디자인에 구현**되어 있어 제안→정식 스펙으로 승격했습니다. 아래 항목은 **개선이 아니라 필수 구현 대상**입니다(기능명세서·DB·기술설계에 반영 완료).

| 기존 제안 | 실제 상태 | 처리 |
|---|---|---|
| IMP-09 댓글·리액션 | 투두 상세에 **댓글** 존재, 라이브 룸에 **응원(이모지)** 존재 | ✅ 정식 스펙化 |
| IMP-10 사진 첨부 | 투두·라이브 룸 **사진 공유** 존재 | ✅ 정식 스펙化 |
| IMP-15 다크 모드 | 설정에 **다크 모드** 존재 | ✅ 정식 스펙化 |
| IMP-21 게이미피케이션 | **라이프/루틴 점수·연속일수·잔디** 핵심 구현 | ✅ 정식 스펙化 |
| IMP-01 라이브 시작 동선 | "내가 할게요"·"지금 바로 시작하기"·라이브 시작 화면 존재 | ✅ 대부분 해소(미세 개선만 잔존) |

또한 v1.0에 **없던** 도메인이 실제 디자인에 있어 신규 항목을 추가합니다(아래 IMP-22~25).

## 1. UX / 사용성

### IMP-01 · 실시간 "라이브" 상태의 시작 방법이 불명확 [P0 · 중]
- **문제**: "서연님이 하는 중 · 라이브"는 보이지만, *내가 어떻게 라이브를 시작/종료*하는지 진입점이 디자인에 없음.
- **제안**: 투두 아이템에 "지금 함" 토글(▶)을 추가하고, 진행 중에는 경과 타이머·"종료" 버튼 노출. 앱을 닫거나 일정 시간(예: 2시간) 무동작 시 자동 종료.
- **근거**: 핵심 차별화 기능인 presence가 *수동 시작*에 의존하면 데이터가 비어 가치가 떨어짐.

### IMP-02 · 빈 상태(Empty State) 설계 부재 [P0 · 하]
- **문제**: 그룹 미가입·활동 없음·투두 없음 상황의 화면이 정의되지 않음.
- **제안**: 각 섹션별 일러스트+CTA형 빈 상태("첫 그룹을 만들어보세요", "급한 일은 없어요 👍").
- **근거**: 신규 사용자의 첫인상·온보딩 이탈 방지.

### IMP-03 · 온보딩 / 첫 그룹 생성 플로우 [P0 · 중]
- **문제**: 가입 직후 빈 홈에 도착할 가능성. 그룹 없는 todly는 가치가 0.
- **제안**: 가입 → "그룹 만들기 or 초대 코드 입력" 강제 분기. 템플릿("이사 준비/여행/가사분담") 제공으로 즉시 가치 체험.
- **근거**: 협업 앱은 *첫 협업 상대 연결*이 활성화의 핵심(Aha moment).

### IMP-04 · "확인이 필요해요"의 인앱 액션 강화 [P1 · 하]
- **문제**: 마감/지연 항목을 보여주지만 즉시 처리 액션(완료·연기·담당변경)이 약함.
- **제안**: 카드에서 스와이프 또는 인라인 버튼으로 "완료/내일로/담당자 변경" 즉시 처리.
- **근거**: 알림→행동 거리를 줄여야 실제 완료율 상승.

### IMP-05 · 색상만으로 긴급도 구분(접근성) [P1 · 하]
- **문제**: 오늘마감(빨강)/지남(주황)을 색으로만 구분 → 색약 사용자 불리.
- **제안**: 색 + 텍스트 뱃지("오늘 마감","1일 지남") + 아이콘 병행(이미 일부 텍스트 있음 → 일관 적용).
- **근거**: WCAG 1.4.1 (색에만 의존 금지).

## 2. 기능 갭 (Feature Gaps)

### IMP-06 · 알림(푸시) 시스템 [P0 · 중]
- **문제**: 마감·멘션·완료 알림 정책 미정의.
- **제안**: 마감 임박/지연, 내게 담당 배정, 멤버 라이브 시작, 그룹 완료 마일스톤 등 이벤트 기반 푸시 + 인앱 알림센터. 사용자별 on/off.
- **근거**: 비동기 협업의 핵심 리텐션 도구.

### IMP-07 · 정산 / 분담(비용) 기능 [P1 · 상]
- **문제**: 이사·여행·가사 시나리오는 *돈 분담*이 동반되는데 현재 없음.
- **제안**: 투두에 비용 필드 + 그룹 정산("누가 얼마 썼고 누가 갚을지") 요약. 차기 핵심 차별화 후보.
- **근거**: "함께 살아가는" 도메인에서 자연스러운 확장(룸메이트/여행 정산 수요 큼).

### IMP-08 · 캘린더 뷰 / 마감 일정 [P1 · 중]
- **문제**: 마감일 기반인데 달력 시각화가 없음.
- **제안**: 그룹/개인 일정을 월·주 캘린더로, 외부 캘린더(구글/애플) 양방향 동기화.

### IMP-09 · 댓글 · 리액션 [P1 · 하]
- **문제**: 투두/활동에 대한 소통 채널 부재.
- **제안**: 투두 단위 댓글·이모지 리액션. presence와 결합해 "함께함" 강화.

### IMP-10 · 첨부(사진/파일) [P2 · 중]
- **제안**: "깨지는 물건 포장" 같은 항목에 참고 사진·영수증 첨부.

### IMP-11 · 검색 / 필터 [P1 · 하]
- **제안**: 투두/그룹/담당자/기한 검색·필터(내 담당만, 미완료만 등).

## 3. 정보 구조 / 네비게이션

### IMP-12 · "그룹" vs "홈" 정보 중복 정리 [P2 · 하]
- **문제**: 홈의 "함께한 진행률"과 그룹 탭의 그룹 목록이 겹침.
- **제안**: 홈=오늘 액션 중심(개인 큐레이션), 그룹=관리/구조 중심으로 역할 명확화.

### IMP-13 · 5개 탭의 인지 부하 [P2 · 하]
- **문제**: 홈/그룹/활동/루틴/프로필 5탭은 다소 많고 "활동↔홈 지금활동중" 개념이 겹칠 수 있음.
- **제안**: 사용 데이터 확인 후 "활동"을 홈 내 탭/세그먼트로 흡수하는 4탭 안 A/B 검토.

## 4. 시각 / 디자인 시스템

### IMP-14 · 디자인 토큰화 [P1 · 하]
- **제안**: 디자인의 블루 그라데이션·프로필 4색·간격·라운드(카드 16~20px)·그림자를 토큰으로 정의해 일관성·다크모드 대비.

### IMP-15 · 다크 모드 [P2 · 중]
- **제안**: 라이트 위주 디자인 → 시스템 연동 다크 테마 추가.

### IMP-16 · 멤버 아바타 이미지 지원 [P2 · 하]
- **문제**: 현재 이니셜+색상만. 
- **제안**: 사진 업로드 옵션(이니셜은 폴백).

## 5. 기술 / 신뢰성 (제품 관점)

### IMP-17 · 오프라인 & 낙관적 업데이트 [P1 · 중]
- **제안**: 투두 토글은 낙관적 UI로 즉시 반영 후 서버 동기화, 실패 시 롤백. 오프라인 캐시.

### IMP-18 · 동시 편집 충돌 처리 [P1 · 중]
- **문제**: 여러 명이 동시에 같은 투두 수정 가능.
- **제안**: 버전 필드(낙관적 락) + 실시간 머지, 충돌 시 사용자 안내.

### IMP-19 · 활동 로그 보존 정책 [P2 · 하]
- **제안**: 활동/presence는 양이 큼 → TTL·아카이브 정책으로 비용·성능 관리.

## 7-B. v2.0 신규 개선 항목 (실제 디자인 기반)

### IMP-22 · 라이브 룸 동시성/지연 처리 [P0 · 상]
- 라이브 룸은 다인원 실시간(참여자·사진·응원)이라 부하·지연·동시성이 핵심. 메시지 순서 보장, 사진 업로드 비동기(썸네일), 재연결 시 상태 복원, 룸 인원 상한·자동 종료(호스트 이탈/무동작) 정책 필요.

### IMP-23 · 게이미피케이션 점수 정의의 투명성 [P1 · 중]
- "라이프 점수/루틴 점수" 산정식이 사용자에게 불투명하면 신뢰가 떨어짐. 점수 획득 규칙(완료·연속·라이브 참여 가중치) 툴팁/도움말 노출, 어뷰징 방지(셀프 완료 반복) 규칙 권장.

### IMP-24 · 친구 프라이버시·차단 [P1 · 중]
- 친구 시스템엔 차단/신고, 검색 노출 범위(@아이디 공개/비공개), 온라인 상태 숨김 옵션이 필요. (DB `friendship_status='blocked'` 반영됨)

### IMP-25 · 사진 저장/모더레이션 [P1 · 상]
- 투두·라이브 룸 사진은 스토리지 비용·부적절 콘텐츠 리스크. 객체스토리지(S3 등)+서명 URL, 용량/포맷 제한, 신고·삭제, 보존정책 필요.

### IMP-26 · "내가 할게요" 충돌 처리 [P2 · 하]
- 두 명이 동시에 "내가 할게요"를 누르는 경우의 담당 확정 규칙(선점/공동담당) 정의.

## 8. 한 줄 총평
디자인은 **"공유 투두 + 실시간 함께함"**이라는 명확한 차별점을 잘 잡았습니다. 가장 시급한 보강은 ① **라이브 상태를 시작하는 명확한 동선**, ② **첫 그룹 연결 온보딩**, ③ **알림 체계**입니다. 이 셋이 갖춰지면 핵심 가치(함께함·책임·완료)가 실제 사용 흐름에서 작동합니다. 중기적으로는 도메인에 자연스러운 **정산·캘린더**가 강력한 확장축이 될 수 있습니다.
