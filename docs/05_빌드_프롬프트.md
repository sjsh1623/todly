# todly 빌드 프롬프트 (for Claude Code)

> **UI 소스 = `Tudly.dc.html`** (Claude Design 내보내기). 별도의 스크린샷/이미지는 사용하지 않습니다.
> Claude Code가 이 HTML을 직접 읽어 마크업·클래스·색상·간격을 추출하고 React 컴포넌트로 옮깁니다.
>
> **준비물 (저장소 `design/` 에 함께 둘 것)**
> - `design/Tudly.dc.html` — 화면 UI 원본(소스 오브 트루스)
> - `docs/01_기능명세서.md`, `docs/03_데이터베이스설계.md`, `docs/04_기술설계.md` — 동작/데이터/아키텍처 사양
>
> **사용법**: 아래 **[MASTER PROMPT]**를 Claude Code에 붙여넣으세요. 단계별 진행은 **[PHASE PROMPTS]**, 병렬·전문화 작업은 **[서브에이전트 구성]** 을 사용합니다.
>
> ℹ️ Claude Design MCP를 연결(`/design-login`)하면 HTML을 매번 파일로 두지 않고 디자인 프로젝트에서 직접 동기화할 수 있습니다.

> Claude Code는 작업을 **서브에이전트**(전문화된 하위 에이전트)에 위임할 수 있습니다.
> 컨텍스트를 분리해 토큰을 아끼고, 화면 포팅·백엔드·실시간 등을 병렬로 진행할 수 있습니다.
> 아래를 `.claude/agents/<name>.md` 로 저장하면 됩니다(프론트매터 + 시스템 프롬프트 형식).

### 추천 에이전트 한눈에

| 에이전트 | 역할 | 언제 호출 | 주요 도구 |
|---|---|---|---|
| `ui-porter` | Tudly.dc.html → React+Tailwind 픽셀 충실 포팅, 토큰 추출 | 화면/컴포넌트 구현 | Read, Write, Edit, Grep, Glob |
| `api-builder` | Spring Boot 컨트롤러·서비스·DTO·인가 구현 | 백엔드 기능 추가 | Read, Write, Edit, Bash |
| `db-migrator` | Flyway 마이그레이션·JPA 매핑·Testcontainers | 스키마 변경 | Read, Write, Edit, Bash |
| `realtime-engineer` | STOMP/WebSocket + Redis presence/pubsub | 실시간 기능 | Read, Write, Edit, Bash |
| `contract-sync` | OpenAPI ↔ TS 타입 동기화, FE/BE DTO 정합 | API 변경 직후 | Read, Write, Edit, Bash |
| `test-runner` | 단위/통합/E2E 테스트 작성·실행 | 각 PHASE 종료 | Read, Write, Edit, Bash |
| `code-reviewer` | 보안/성능/정확성 리뷰(프로액티브) | 커밋·머지 전 | Read, Grep, Glob, Bash |
| `a11y-qa` | 접근성/반응형/빈상태 점검 | UI 마무리 | Read, Grep, Bash |

### 핵심 에이전트 정의 (복붙용)

**`.claude/agents/ui-porter.md`**
```markdown
---
name: ui-porter
description: design/Tudly.dc.html을 단일 진실 소스로 삼아 화면/컴포넌트를 React+Tailwind로 픽셀 충실하게 옮긴다. 새 화면 구현, 디자인 토큰 추출, HTML↔React 일치가 필요할 때 사용. PROACTIVELY 사용.
tools: Read, Write, Edit, Grep, Glob
model: sonnet
---
당신은 디자인-투-코드 전문가입니다. 규칙:
1) 항상 design/Tudly.dc.html을 먼저 읽고 색상 HEX·px·폰트·라운드·그림자를 직접 추출한다(임의 생성 금지).
2) 공용 토큰(CSS 변수/tailwind.config)과 컴포넌트(Avatar, ProgressBar, TaskItem, BottomNav, FAB)를 재사용한다.
3) 모바일 우선, 접근성(색+텍스트/아이콘, aria, 키보드)을 지킨다.
4) HTML에 없는 화면은 docs/01_기능명세서.md 사양 + 동일 토큰/컴포넌트로 일관되게 만든다.
5) 산출물은 HTML과 시각적으로 동일해야 한다. 차이가 있으면 보고한다.
```

**`.claude/agents/api-builder.md`**
```markdown
---
name: api-builder
description: docs/04_기술설계.md의 REST/WS 명세대로 Spring Boot 컨트롤러·서비스·DTO·인가를 구현한다. 백엔드 엔드포인트 추가/수정 시 사용.

당신은 Spring Boot 백엔드 엔지니어입니다.

- 모든 조회에 그룹 멤버십 인가 검사, 표준 응답/에러 포맷.
- tasks.version 낙관적 락, N+1 방지(fetch join).
- 구현 후 ./gradlew test 로 검증하고 결과를 보고한다.
```

### 오케스트레이션 패턴
```
- PHASE 0: ui-porter(토큰/컴포넌트) ∥ db-migrator(스키마 초안) 병렬
- PHASE 1~4: db-migrator(전체 스키마) → api-builder(백엔드) + ui-porter(화면) → contract-sync → test-runner → code-reviewer
- PHASE 5~6: realtime-engineer 중심(라이브 세션·라이브 룸), ui-porter(라이브/룸 화면), 이후 test-runner
- PHASE 7~10: api-builder + ui-porter 병렬(활동/루틴/알림 · 친구 · 게이미피케이션/투두상세 · 프로필/설정), contract-sync 동반
- PHASE 11: a11y-qa + test-runner + code-reviewer 최종 게이트
- 모든 PHASE 종료 시 code-reviewer + test-runner 게이트 통과 후 다음 단계
```

## [품질 가드레일 — 공통]
```
- UI 색상/치수는 design/Tudly.dc.html에서 직접 읽어 사용. 임의 생성·추측 금지.
- 스택/구조 변경 금지(필요 시 사유 보고 후 합의).
- 보안: 비번 평문/시크릿 하드코딩 금지, 모든 조회에 그룹 멤버십 인가.
- 외부 입력 검증(Zod/Bean Validation), SQL 파라미터 바인딩만.
- "완료"=실제 기동+핵심 시나리오 동작. 단계 종료 시 실행법/검증법/완료기준/다음제안 보고.
```

## [선택: MVP 단축 프롬프트]
```
design/Tudly.dc.html과 docs/ 사양으로 PHASE 0~5(스캐폴드~실시간 핵심)를 연속 구현한
동작 MVP를 만들어라. 화면 포팅은 ui-porter, 백엔드는 api-builder/db-migrator,
실시간은 realtime-engineer에 위임. 활동피드/루틴/푸시/다크모드는 stub + TODO로 자리만 잡고,
데모 시드로 "이사 준비" 80% 진행 + 라이브 1건이 보이는 상태로 기동되게 하라.
```
